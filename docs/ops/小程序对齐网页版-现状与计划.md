## 二、功能差距矩阵

> 网页版 pages：community, home, lostfound, **messages**, news, planner, profile
> 小程序 pages：home, community, planner, compose, news, lostfound, profile, login
> 后端 routers（全齐）：auth, courses, lostfound, **messages**, news, posts, **push**, **upload**, users

| 功能 | 网页版 | 小程序 | 后端 API | 差距性质 | 前端改动量 |
|------|:---:|:---:|----------|----------|:---:|
| 首页信息流 | ✅ | ✅ | posts ✅ | 脏数据 | 无 |
| 社区浏览/发帖 | ✅ | ✅ compose | posts ✅ | 标题 bug + i18n | 小 |
| 校园新闻 | ✅ | ✅ | news ✅ | **数据空** | 无 |
| 失物招领 | ✅ | ✅ | lostfound ✅ | **数据空** | 无 |
| 选课规划 | ✅ C3 完整 | ❌ mock | **courses ✅ C3 现成** | 没接 API | 中 |
| 图片上传 | ✅ | ❌ | **upload ✅ 现成** | 没接 | 小 |
| 评论 | ✅ | ❌ toast 占位 | posts（待核实） | 缺功能 | 大 |
| 私信 / WebSocket | ✅ messages | ❌ 无页 | **messages ✅ 现成** | 缺页 | 大 |
| 资料编辑 | ✅ | 部分 | users ✅ | 编辑入口缺 | 小 |
| 注册入口 | ✅ | ❌ | auth(email/register) ✅ | 缺入口 | 中 |
| 推送通知 | ✅ | ❌ | push ✅ | 没接 | 中 |

**关键洞察**：私信、图片上传、选课规划、推送 —— **后端 API 全部就绪**，小程序只是没接。这是"少动前端"策略成立的基础。

## 三、对齐策略（按"少动前端"分 Tier）

### 🟢 Tier 1 — 零前端改动（纯后端/数据，应最先做）
1. **补新闻 seed 数据**（生产 Supabase）→ 新闻页立刻有内容
2. **补失物招领 seed 数据** → 失物招领页立刻有内容
3. **清测试垃圾帖**（`asd`/`qwe`、E2E Bot、微信测试号）→ 上线前必修

### 🟡 Tier 2 — 小前端改动（接现成 API + 修 bug）
4. **选课规划接 C3**：`planner.js` 接 `courses.py` 的 C3 API（web 端 planner 已实现，可参照 `frontend/js/pages/planner.js`）
5. **图片上传**：`compose.js` / `lostfound.js` 接 `upload.py`（发帖带图、失物招领带图）
6. **修社区标题 bug**：`community.js` 卡片渲染（P2）
7. **补 i18n category 映射**：`utils/i18n.js` 加 `discussion` → 中文（P1）

### 🔴 Tier 3 — 大前端改动（新页面/组件，谨慎评估）
8. **私信系统**：新页 + WebSocket（后端 `messages.py` + `websocket_manager.py` 现成）
9. **评论**：新组件/页（核实 `posts.py` 评论端点后定）
10. **注册入口**：login 页加注册（后端 `auth/email/register` 现成）

---

## 四、下一步建议顺序

1. **先清数据 + 补 seed**（Tier 1）—— 见效最快，零风险动前端
2. **选课规划接 C3**（Tier 2 #4）—— 核心功能，API 现成，收益最大
3. **修 bug + i18n**（Tier 2 #6 #7）—— 小改快修
4. **图片上传**（Tier 2 #5）
5. Tier 3 按需评估（私信/评论/注册）

**协作提示**：选课规划属「学术规划」模块（SYF），私信属「私信系统」（SYF），改前按 module-registry 通知负责人。后端 API 补充若动共享文件（main/database/models）须群内通知。

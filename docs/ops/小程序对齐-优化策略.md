# 小程序端优化策略（基于三端代码通读）

> 2026-06-25，基于小程序端 / 网页端 / 后端 API 三路代码通读产出。
> 配套文档：`docs/ops/小程序对齐网页版-现状与计划.md`（测试现状与差距矩阵）。
> 原则：**可动小程序前端对齐功能，但严守小程序现有视觉风格**（设计 token 见 §四）。

## 一、通读后的关键认知更新（修正之前的判断）

这几条很重要，直接决定策略：

1. **C3 选课规划：后端没有聚合端点**。网页版 planner（1426 行）的毕业进度计算 `_calcGraduationProgress`、智能推荐 `_getRecommendations`、前置课解析——**全部在前端**，后端只存课程元数据 + 个人进度（`/courses` + `/courses/progress/me`）。专业数据硬编码在网页 `js/data/programmes.js`，**且只有 DSAI 一个专业完整**，CS/网安是 `comingSoon` 占位。
   → 小程序要对齐 C3，**要么移植 programmes.js + 算法到小程序，要么新增后端端点**（动 courses.py = SYF 模块，需协调）。

2. **图片上传比想象简单**：`POST /upload?module=` 返回**绝对 URL**（Supabase Storage，`https://...supabase.co/storage/v1/object/public/uploads/...`），前端无需拼接。小程序用 `wx.uploadFile`（注意：不是 `wx.request`）。网页端帖子**不支持配图**（只有 lostfound 支持）——小程序可以补上，移动端发帖带图是刚需。

3. **私信 WS 协议成熟可复用**：`ws(s)://host/api/v1/messages/ws?token=<jwt>`（token 走 query），消息格式 `{type:"chat"|"ping"|"mark_read", ...}`。小程序用 `wx.connectSocket` + 同协议。网页端 nav **没有未读角标**、在线状态是本地假判断——小程序可以做更好。

4. **评论端点齐全**：posts 和 news 各一套评论 API，`CommentOut` 含 author 信息，单层平铺无嵌套。小程序只需补 UI。

5. **app.wxss 是 5 层叠加的技术债**（前 4 层死代码占 1900 行，最后一层 1899+ 才生效）。改样式极易引用到废弃值——这是**最大的维护风险**，动前端前必须处理或建 token 速查表。

6. **community 标题 bug**：前端代码与 home **逐字等价**（`normalizePost`/wxml/CSS 都一致），所以根因**不在小程序前端**，极可能是该帖 `item.title` 数据本身为空 / 是不同的帖子。**待抓包确认**（home 和 community 的 page_size/sort 不同，可能取到不同帖子）。

7. **home 与 community 大量重复代码**（avatarUrl/compactNumber/normalizePost/toggleLike 等），应抽 utils。

## 三、优化策略（Phase 0–4）

> 工作量：S=半天内，M=1–2 天，L=3+ 天。每项标注【接 API】【改文件】【保风格】【工作量】。

### Phase 0 — 技术债 + bug（先做，降后续风险）

| # | 事项 | 细节 | 工作量 |
|---|------|------|:--:|
| 0.1 | **建设计 token 速查表 + 隔离 app.wxss 死代码** | 把 1899+ 的生效值抽成 `styles/tokens.wxss`（或注释锚点），新增样式只引 token；可选：把前 4 层死代码移到 `app.legacy.wxss` 备份 | M |
| 0.2 | **确认+修 community 标题 bug** | 抓包 `/posts` 看该帖 title 字段；若数据空→查 posts 表；若字段有值→查 `buildVisiblePosts` 过滤逻辑 | S |
| 0.3 | **修 i18n 繁体漏翻** | `i18n.js` zh-Hant 的 lostfound/news/planner/profile 用 Object.assign 继承简体，补繁体字 | S |
| 0.4 | **抽 home/community 重复代码** | normalizePost/compactNumber/avatarUrl/toggleLike → `utils/post.js` | S |
| 0.5 | **修 compose 跳转** | 发帖成功回 community 而非 home（`switchTab` community） | S |
| 0.6 | **清 mock 残留文案** | home.statusCopy、planner.reminderCopy 仍写"UI 预览/mock"，与已接 API 矛盾 | S |

### Phase 1 — 接现成 API（中工作量、高收益）

【保风格】所有新 UI 复用 §四 token + 列表页骨架模板。

| # | 事项 | 接 API | 改/加文件 | 工作量 |
|---|------|--------|----------|:--:|
| 1.1 | **帖子配图** | `wx.uploadFile → POST /upload?module=posts` → 拿 url → `POST /posts {title,content,category}` 不直接传图（图走 upload 存 url，正文用 markdown img 或新增 image_url）⚠️需确认 posts 是否支持图字段 | `compose.js/wxml` 加图片选择器 | M |
| 1.2 | **失物招领配图** | `POST /upload?module=lostfound` → `POST /lostfound {image_url}` | lostfound 发表表单（目前小程序 lostfound 似乎只能浏览，需确认能否发表）| M |
| 1.3 | **评论（posts）** | `GET /posts/{id}/comments` + `POST /posts/{id}/comments` | community 帖子卡加评论展开 + 输入；或新建 comment 组件 | M |
| 1.4 | **资料编辑** | `PUT /users/me` + `POST /upload?module=avatars` → `PUT /users/me {avatar_url}` | profile 加编辑弹层 | M |
| 1.5 | **注册入口** | `POST /auth/email/register` | login 页加"注册"segment | S |

### Phase 2 — 选课规划 C3（最大功能块）

**决策点（需你拍板）**：
- **方案 A（推荐）**：移植网页 `programmes.js` + `_calcGraduationProgress` + `_getRecommendations` 到小程序。不改后端，不碰 SYF 模块，先做 DSAI。**工作量 L**。
- **方案 B**：新增后端 `GET /courses/programmes` + `GET /courses/graduation-status` 聚合端点，小程序只渲染。架构更干净，但要改 courses.py（SYF 模块，需通知+PR review），且数据仍只有 DSAI。**工作量 L+**。

【接 API】`GET /courses?page_size=50` + `GET /courses/progress/me` + `PUT /courses/progress`
【改文件】`planner.js/wxml` 整页重写（当前是硬编码 mock）
【保风格】复用网页的圆形进度环思路，但用小程序 canvas 或 SVG（注意之前 C3 有过 SVG className crash，见 commit 94f841d）；卡片用 feed-card/utility-card
【约束】只有 DSAI 专业有数据，CS/网安显示"敬请期待"

### Phase 3 — 私信（最大工作量，新建页）

【接 API】`GET /messages/conversations` + `GET /messages/history/{id}` + `POST /messages/{id}` + `GET /messages/unread-count` + WS `/messages/ws?token=`
【改文件】**新建** `pages/messages/`（js/wxml/wxss/json）+ 加 tabBar 项 or profile 入口 + 未读角标
【保风格】会话列表用 feed-card 变体，聊天页用 bubble 样式（需新增，参照 iMessage 风格但配色用 token）
【技术点】`wx.connectSocket` + 30s 心跳 + 断线 REST 降级 + 小程序前后台生命周期处理（onHide 断连、onShow 重连）
【工作量】L+（小程序 WS 生命周期比网页麻烦）

### Phase 4 — 数据（需你授权改生产 Supabase）

- 补 news seed（`POST /news` 需 admin）
- 补 lostfound seed
- 清测试垃圾帖（asd/qwe、E2E Bot、WeChat User 1232）

## 四、风格一致性基准（设计 token 速查）

> 新增任何 UI **只能用这些值**。来源：app.wxss:1899+（Premium campus refresh 层）。

**颜色**
| 用途 | hex |
|------|-----|
| 页面背景 | `#F7FAF6` |
| 主文字 | `#142033` |
| 次要文字 | `#6B7280` |
| 主按钮/CTA（蓝） | `#0F8FBE`（阴影 `rgba(15,143,190,0.16)`）|
| 次按钮/绿 | `#3D7A2E` |
| 强调/亮绿 | `#7FBF4D` |
| 卡片 | 白 `#FFFFFF` + 边 `1rpx solid rgba(90,130,90,0.12)` + 阴影 `0 16rpx 48rpx rgba(30,70,40,0.06)` |
| 分段器底 | `#EEF5EC`，active 白底绿字 |
| hero 渐变 | `linear-gradient(135deg,#7FBF4D,#6DB344)` |
| tabBar | selectedColor `#3D7A2E`，color `#667085` |

**字号**：app-title 60rpx/700、feed-title 36rpx/700、feed-body 30rpx、subtitle 27rpx、eyebrow 21rpx/700 uppercase。字体栈 `-apple-system, "SF Pro Text", "Segoe UI"`。

**复用 class**（不要新造，直接用）
- 卡片：`feed-card` / `article-card` / `utility-card` / `compose-card` / `form-sheet` / `empty-state`
- 按钮：`primary-action`（蓝）/ `secondary-action`（白底绿字）/ `danger-action`（红）/ `wechat-action`
- 容器：`page-shell`、`top-bar`、`search-dock`、`segmented`、`brand-lockup`、`eyebrow`、`app-title`、`subtitle`

**新页骨架模板**（所有列表/表单页照这个套）
```
<app-header bind:languagechange="handleLanguageChange" />
<view class="page-shell">
  <view class="top-bar"> <view class="brand-lockup">
    <view class="eyebrow">{{text.eyebrow}}</view>
    <view class="app-title">{{text.title}}</view>
    <view class="subtitle">{{text.subtitle}}</view>
  </view></view>
  <view class="search-dock">...</view>        <!-- 可选 -->
  <view class="segmented">...</view>          <!-- 可选筛选 -->
  <view wx:if="{{!list.length && !loading}}" class="empty-state">...</view>
  <view wx:for="{{list}}" class="feed-card">...</view>
</view>
```

**i18n**：`data.text = getTexts(scope)` 初始化 + wxml `{{text.xxx}}`；切换走 app-header 的 `languagechange` → `applyLocale`。

## 五、建议实施顺序与里程碑

1. **M1（打底）**：Phase 0 全部 —— 清技术债、修 bug、抽公共代码。让后续改动安全。
2. **M2（功能补齐-小）**：Phase 1（配图、评论、资料编辑、注册）—— 接现成 API，见效快。
3. **M3（核心功能）**：Phase 2 C3 选课规划 —— 移植 DSAI（方案 A）。
4. **M4（社交）**：Phase 3 私信 —— 新页 + WS。
5. **随时（需授权）**：Phase 4 数据。

每个 Phase 独立可交付、可测、可回滚。
